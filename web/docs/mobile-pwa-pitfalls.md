# Mobile & PWA Pitfalls

Gotchas hit while building the mobile shell, pull-to-refresh, and the iOS
home-screen experience — kept here so the next person (or agent) doesn't
re-debug the same things.

---

## 1. Bun dev server won't rebuild Tailwind on its own

`web/src/tailwind-out.css` is a **static generated file**, not compiled
on-the-fly. `bun run dev` runs `tailwindcss --watch` *alongside* `bun --hot`
in the same script — if that dev script isn't the thing actually running
(e.g. you started `bun src/index.ts` directly, or the watcher process died),
new Tailwind classes silently do nothing. The page still renders — just
without the new utility, which looks like "my class isn't working" rather
than "the CSS was never rebuilt."

**Symptom:** a component you just styled renders unstyled/collapsed (e.g. a
chart container with no height, a nav with no spacing).

**Fix:** rebuild once before testing —

```bash
cd web && bunx tailwindcss -i src/index.css -o src/tailwind-out.css
```

For active dev, make sure `bun run dev` (not a bare `bun src/index.ts`) is
the process actually running.

## 2. Bun's dev server doesn't always pick up brand-new files

Editing an existing file hot-reloads instantly. A **newly created** file
that something already imports can 500 with `Could not resolve` until the
watcher notices it — a `touch path/to/new-file.ts` (or one more save) is
enough to kick it. Not a code bug, just a dev-server quirk; doesn't happen
in production builds since the container starts fresh.

## 3. Bun's HTML bundler only rewrites *known* `<link rel>` values

`<link rel="icon">`, `rel="stylesheet"`, `rel="apple-touch-icon"` etc. get
their relative `href` rewritten to a hashed `/_bun/asset/...` path
automatically. `rel="apple-touch-startup-image"` is not on that list — Bun
leaves the raw relative path untouched, which then resolves wrong (the
SPA's `"/*"` catch-all serves `index.html` instead of the image).

**Fix pattern:** use an absolute path (`/splash/foo.png`) and add an
explicit static route in `src/index.ts`:

```ts
"/splash/:file": (req) =>
  new Response(Bun.file(`./src/splash/${req.params.file}`)),
```

Same reasoning applies to `manifest.json` and `service-worker.js` — both
are referenced from *runtime JS* (`register-sw.ts`), never from an
`index.html` `<link>`, so the bundler never sees them either.

## 4. iOS ignores the Web App Manifest for install/splash

Android/Chrome build the "Add to Home Screen" install prompt and splash
screen entirely from `manifest.json` (`icons`, `background_color`,
`display`). iOS Safari **does not read the manifest for any of that** — it
only looks at:

```html
<meta name="apple-mobile-web-app-capable" content="yes" />
<meta name="apple-mobile-web-app-status-bar-style" content="black-translucent" />
<meta name="apple-mobile-web-app-title" content="CRM Wika" />
<link rel="apple-touch-icon" href="..." />
<link rel="apple-touch-startup-image" ... />
```

Without `apple-mobile-web-app-capable` specifically, iOS won't even treat
the home-screen shortcut as a standalone app — it just opens Safari with
full browser chrome. This is why "it works on Android" tells you nothing
about iOS; test both explicitly.

## 5. `apple-touch-startup-image` needs an *exact* device match

A single `apple-touch-startup-image` with only
`media="(prefers-color-scheme: dark)"` and no device dimensions is
**silently ignored on real iOS hardware** — even though it's valid HTML and
"looks" like it should work. iOS requires the media query to match the
device's exact `device-width` / `device-height` / `-webkit-device-pixel-ratio`
(CSS pixels, not physical pixels) before it'll pick an image at all;
`prefers-color-scheme` only further narrows *within* an already-matched
device query, it doesn't stand alone.

**Fix:** ship one `<link>` per device-size bucket × color scheme, e.g.:

```html
<link
  rel="apple-touch-startup-image"
  media="screen and (device-width: 390px) and (device-height: 844px)
         and (-webkit-device-pixel-ratio: 3) and (orientation: portrait)
         and (prefers-color-scheme: light)"
  href="/splash/splash-390x844-3-light.png"
/>
```

`web/src/index.html` currently covers 8 common iPhone sizes (SE through Pro
Max) × light/dark = 16 images. No image toolchain exists in this project —
they were generated by rendering `logo.svg` centered on a solid background
in headless Chromium (Playwright) at each device's *physical* pixel size
(`device-width × dpr`, `device-height × dpr`) and screenshotting. Regenerate
the same way if the logo changes.

**Also note:** this follows the **device's system appearance** (Settings →
Display), not the app's own in-app theme toggle — there's no way to read
app/localStorage state before any JS has run, so a device in system dark
mode briefly shows the dark splash even though the app itself always boots
in light theme.

**This can't be tested by an agent.** The splash only renders during the
native "launch from home screen" transition — no browser-automation tool
(Playwright et al.) can simulate that; it requires a real device, remove
+ re-add the home-screen icon after every change (iOS caches the config
from whenever it was installed).

## 6. Mobile layout must use `h-dvh`, not `min-h-dvh`

`AppLayout`'s mobile wrapper (`app-layout.tsx`) needs a **fixed** viewport
height so `<main>`'s `flex-1 overflow-auto` actually clips and becomes the
scroll container:

```tsx
// wrong — grows past the viewport, so the *document* scrolls instead of <main>
<div className="flex min-h-dvh flex-col">

// right — caps at viewport height, <main> scrolls internally
<div className="flex h-dvh flex-col">
```

With `min-h-dvh`, `main.scrollTop` is **always 0** no matter where the user
has actually scrolled, because the whole page/document is what's scrolling.
Anything that reads `scrollTop` to make a decision (pull-to-refresh's "am I
at the top" check, scroll-spy, sticky-header logic, etc.) silently breaks —
it thinks it's always at the top. This one cost a full round-trip because
it *looked* like a gesture-handling bug when it was actually a layout bug.

## 7. Pull-to-refresh: touch gesture handling gotchas

`web/src/hooks/use-pull-to-refresh.ts` — reusable if another mobile-only
gesture is needed later. Key things that weren't obvious up front:

- **Handle `touchcancel`, not just `touchend`.** iOS fires `touchcancel`
  instead of `touchend` when it hands a gesture off mid-stream (native
  rubber-band, a system gesture taking over). Without a `touchcancel`
  listener resetting state, the pull indicator/translateY can get stuck
  permanently mid-gesture.
- **Abort tracking entirely on direction reversal**, don't just zero the
  distance. If you only reset the *distance* but keep the original
  `startY`, a finger that dips down again mid-gesture computes delta from
  the stale start point and jumps.
- **A small dead zone (~10px) before calling `preventDefault()`** avoids
  hijacking a light touch-and-adjust at the very top of the page.
- **Gesture math lives in refs, not state.** `touchmove`/`touchend` need
  the *latest* value on every event; putting the running pull-distance in
  `useState` and depending on it from the `useEffect` would re-bind the
  listeners on every frame. State is only used for what actually needs to
  trigger a re-render (the indicator's height/opacity).

## 8. TypeScript narrowing doesn't survive into `function` declarations

```ts
useEffect(() => {
  const el = containerRef.current;
  if (!el) return;

  function onTouchStart() {
    el.scrollTop; // ❌ TS: 'el' is possibly 'null'
  }
}, []);
```

`el` is narrowed to non-null right above, but TS doesn't carry that into a
**hoisted function declaration** — only into function *expressions*
(arrow functions) defined after the check:

```ts
const onTouchStart = () => {
  el.scrollTop; // ✅ fine, el is captured post-narrowing
};
```

Prefer `const fn = () => {}` over `function fn() {}` for event handlers
declared inside an effect that close over a narrowed outer variable.
